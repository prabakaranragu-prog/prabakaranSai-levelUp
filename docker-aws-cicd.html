What is Docker?

Package your Spring Boot app once — run it anywhere, without "works on my machine" problems

Real World Analogy — Shipping Container

Before Docker: Dev says "it works on my machine!" → Prod server crashes because Java version different, configs missing. Docker = a shipping container — pack your app + all its dependencies (Java 21, Spring Boot JARs, configs) into one container box. Ship it to any server — it runs exactly the same. No surprises!

5 Core Docker Concepts

01
🖼️
Docker Image
A read-only blueprint/template of your app. Like a class in Java — you define it once. Contains: OS layer + Java runtime + your JAR file + configs.

Built from a Dockerfile. Stored in Docker Registry (Docker Hub / AWS ECR).
02
📦
Docker Container
A running instance of an Image. Like an object created from a class.
You can run 10 containers from the same image — they run independently.

Analogy: Image = class definition. Container = new MyApp() running in memory.
03
📄
Dockerfile
A text recipe file that tells Docker how to build your Image. Step by step instructions — which base image to use, copy your JAR, set port, run command.

Think: like a pom.xml — you define, Docker builds.
04
🏪
Docker Registry
A storage hub for Docker Images. Push your built image here, pull from anywhere.
• Docker Hub — Public registry (free)
• AWS ECR — Private registry for AWS deployments

Like Maven Central but for Docker images.
05
🎵
Docker Compose
A YAML file to run multiple containers together. Your Spring Boot app needs MySQL + Redis? Define all 3 in one docker-compose.yml — run with one command.

docker compose up → spins everything up automatically.
06
💾
Docker Volume
Containers are stateless by default — data is lost when container stops. Volumes = persistent storage mounted into containers.
MySQL data → save to volume → restart container → data still there!

Like a USB drive attached to your container.

Writing a Dockerfile

Turn your Spring Boot JAR into a Docker Image — step by step recipe

Basic Dockerfile for Spring Boot

  Dockerfile
# ───────────────────────────────────────────────────────────── # STEP 1: Base Image — Start from official Java 21 image # Think: "I want a server that already has Java 21 installed" # ───────────────────────────────────────────────────────────── FROM eclipse-temurin:21-jre-alpine # STEP 2: Set working directory inside the container # Like: cd /app inside the container WORKDIR /app # STEP 3: Copy your compiled JAR into the container # target/myapp.jar → /app/app.jar COPY target/sonata-app.jar app.jar # STEP 4: Tell Docker which port your Spring Boot app uses # (Documentation only — doesn't actually open the port) EXPOSE 8080 # STEP 5: Command to run when container starts # Same as: java -jar app.jar in terminal ENTRYPOINT ["java", "-jar", "app.jar"]

Build & Run Commands

1
Build your Spring Boot JAR first
Run Maven to compile and package: mvn clean package -DskipTests
This creates target/sonata-app.jar
2
Build Docker Image from Dockerfile
docker build -t sonata-app:1.0 .
-t = tag (name:version)  |  . = look for Dockerfile in current directory
3
Verify the image was created
docker images → you'll see sonata-app listed with size ~200MB
4
Run a Container from the Image
docker run -d -p 8080:8080 --name my-api sonata-app:1.0
-d = detached (background)  |  -p 8080:8080 = host:container port mapping
Now open browser: http://localhost:8080 — your Spring Boot app is running!
5
Useful Commands to manage containers
docker ps → list running containers
docker logs my-api → see Spring Boot logs
docker stop my-api → stop container
docker rm my-api → remove container
docker exec -it my-api bash → get shell inside container

Production-Grade: Multi-Stage Dockerfile

Why Multi-Stage?

Stage 1: Use a big JDK image to compile/build your JAR. Stage 2: Use a tiny JRE image to just run the JAR. Result: Final image is much smaller (200MB vs 600MB) — faster to push/pull in production!

  Dockerfile (Multi-Stage — Production Ready)
# ── STAGE 1: Build Stage ───────────────────────────────────── # Use full JDK to compile the Maven project FROM maven:3.9-eclipse-temurin-21 AS builder WORKDIR /build # Copy pom.xml first (Docker caches this layer if unchanged → faster rebuilds!) COPY pom.xml . RUN mvn dependency:go-offline -q # Copy source code and build COPY src ./src RUN mvn clean package -DskipTests -q # ── STAGE 2: Runtime Stage ─────────────────────────────────── # Use tiny JRE Alpine (only ~50MB!) to run the app FROM eclipse-temurin:21-jre-alpine WORKDIR /app # Create non-root user for security (never run as root in prod!) RUN addgroup -S appgroup && adduser -S appuser -G appgroup USER appuser # Copy ONLY the built JAR from Stage 1 (not the whole Maven/JDK!) COPY --from=builder /build/target/*.jar app.jar # JVM tuning for containers — tell JVM to respect container memory limits ENV JAVA_OPTS="-XX:+UseContainerSupport -XX:MaxRAMPercentage=75.0" EXPOSE 8080 ENTRYPOINT ["sh", "-c", "java $JAVA_OPTS -jar app.jar"]

Docker Compose

Run Spring Boot + MySQL + Redis all together with one command

When do you use Docker Compose?

Your Spring Boot app depends on MySQL, Redis, and Kafka. Instead of running 4 separate docker run commands and manually connecting them — write ONE docker-compose.yml and run docker compose up. All services start together, on the same network, talking to each other!

  docker-compose.yml — Spring Boot + MySQL + Redis
# Docker Compose version version: '3.9' services: # ── 1. Your Spring Boot Application ────────────────────────── sonata-api: build: . # Build image from local Dockerfile container_name: sonata-api ports: - "8080:8080" # localhost:8080 → container:8080 environment: # Pass DB connection as env vars (not hardcoded in app!) - SPRING_DATASOURCE_URL=jdbc:mysql://mysql-db:3306/sonatadb - SPRING_DATASOURCE_USERNAME=root - SPRING_DATASOURCE_PASSWORD=secret123 - SPRING_REDIS_HOST=redis-cache - SPRING_REDIS_PORT=6379 depends_on: mysql-db: condition: service_healthy # Wait for MySQL to be ready! redis-cache: condition: service_started networks: - sonata-network restart: on-failure # ── 2. MySQL Database ───────────────────────────────────────── mysql-db: image: mysql:8.0 # Pull official MySQL image container_name: mysql-db environment: - MYSQL_ROOT_PASSWORD=secret123 - MYSQL_DATABASE=sonatadb ports: - "3306:3306" # Connect from local DB tools like DBeaver volumes: - mysql-data:/var/lib/mysql # Persist data even after container restart! healthcheck: # Check if MySQL is actually ready test: ["CMD", "mysqladmin", "ping", "-h", "localhost"] interval: 10s timeout: 5s retries: 5 networks: - sonata-network # ── 3. Redis Cache ──────────────────────────────────────────── redis-cache: image: redis:7-alpine container_name: redis-cache ports: - "6379:6379" networks: - sonata-network # Named volumes — data stored by Docker, survives container restarts volumes: mysql-data: # Custom network — all services can talk to each other by service name networks: sonata-network: driver: bridge
▶️
Start Everything
docker compose up -d

Starts all 3 services (API + MySQL + Redis) in background. Spring Boot automatically connects to MySQL!
📋
View Logs
docker compose logs -f sonata-api

Follow live logs of your Spring Boot app. Great for debugging startup issues!
🔄
Rebuild After Code Change
docker compose up -d --build

Rebuilds the image with your latest code changes and restarts the container.
⏹️
Stop & Clean Up
docker compose down -v

Stops all containers and removes volumes. Warning: -v deletes data too!

AWS — Amazon Web Services

The cloud platform where you deploy your Dockerized Spring Boot app to the world

What is AWS? Simple Explanation

Imagine you need a server to host your Spring Boot app. Instead of buying a physical server (expensive!), you rent compute power, storage, and networking from Amazon. AWS has 200+ services — but as a backend developer you only need to know ~10 of them to deploy production-grade apps.

 AWS GLOBAL INFRASTRUCTURE 

AWS Cloud
├── Region (e.g., ap-south-1 Mumbai)
│   ├── Availability Zone 1 (AZ-1a)  — Physical Data Center
│   ├── Availability Zone 2 (AZ-1b)  — Different building
│   └── Availability Zone 3 (AZ-1c)  — Yet another building
│
├── Region (us-east-1 N.Virginia)
│   └── ...more AZs
│
└── 33+ Regions globally

Why multiple AZs? High Availability!
If AZ-1a data center has fire → AZ-1b still serves your users!

Core AWS Services for Backend Developers

🖥️
EC2
Elastic Compute Cloud. A virtual server in the cloud. Deploy your JAR or Docker container on it. You choose CPU, RAM, OS.
Compute
🗄️
S3
Simple Storage Service. Store files, images, videos, JAR files, backups. Like Google Drive but for apps. Pay per GB stored.
Storage
🐳
ECS
Elastic Container Service. Run Docker containers on AWS. You define task (container config), ECS manages scaling, restart, load balancing.
Containers
📦
ECR
Elastic Container Registry. Private Docker Hub for AWS. Store your Spring Boot Docker images here. ECS pulls from ECR to deploy.
Registry
🗃️
RDS
Relational Database Service. Managed MySQL/PostgreSQL. No need to manage server — AWS handles backups, patches, replication.
Database
⚖️
ALB
Application Load Balancer. Distributes incoming HTTP traffic across multiple EC2/ECS instances. Handles 1M users — auto routes to healthy instances.
Networking
🔐
IAM
Identity & Access Management. Control who/what can access your AWS resources. Users, Roles, Policies. Principle of least privilege.
Security
🌐
VPC
Virtual Private Cloud. Your own isolated network in AWS. Public subnets (accessible from internet), Private subnets (only internal access — put DB here!).
Networking
⚡
Lambda
Serverless functions. Run Java code without a server. Pay per execution. Great for event-driven tasks like S3 file processing, SQS message handling.
Serverless
🔑
Secrets Manager
Store DB passwords, API keys securely. Your Spring Boot app fetches secrets at runtime — no hardcoded passwords in code!
Security
📊
CloudWatch
Monitoring & Logs. View your Spring Boot app logs, set CPU/memory alerts, create dashboards. Like Grafana but built into AWS.
Monitoring
📨
SQS / SNS
Message Queue (SQS) and Notification (SNS) services. Decouple microservices. Spring Boot publishes to SQS, another service consumes asynchronously.
Messaging

EC2 — Deploy Spring Boot on a Virtual Server

Step-by-step: Launch EC2, install Docker, run your containerized Spring Boot app

1
Launch EC2 Instance (AWS Console)
Go to AWS Console → EC2 → Launch Instance
• AMI: Amazon Linux 2023 (free tier eligible)
• Instance Type: t2.micro (free tier) or t3.small for production
• Key Pair: Create a .pem file — this is your SSH login key (save it safely!)
• Security Group: Allow port 22 (SSH) and 8080 (your app)
2
SSH into your EC2 Instance
chmod 400 my-key.pem
ssh -i my-key.pem ec2-user@your-ec2-public-ip
You're now inside the server! It's a fresh Linux machine.
3
Install Docker on EC2
sudo yum update -y
sudo yum install -y docker
sudo systemctl start docker
sudo systemctl enable docker
sudo usermod -aG docker ec2-user
exit  then SSH back in — now you can run docker without sudo
4
Pull your Docker Image and Run
docker pull your-dockerhub-username/sonata-app:latest
docker run -d -p 8080:8080 --name sonata your-dockerhub-username/sonata-app:latest
Now access: http://your-ec2-public-ip:8080/api/health — Live in production!
5
View Logs & Monitor
docker logs -f sonata → Live Spring Boot logs
Or check AWS CloudWatch → Log Groups → Your app logs (if configured)

Push Docker Image to AWS ECR

  Push to ECR — Terminal Commands
# Step 1: Authenticate Docker to your ECR registry # Replace ACCOUNT_ID and REGION with your values aws ecr get-login-password --region ap-south-1 | docker login --username AWS --password-stdin 123456789.dkr.ecr.ap-south-1.amazonaws.com # Step 2: Create ECR Repository (one time only) aws ecr create-repository --repository-name sonata-app --region ap-south-1 # Step 3: Build your Docker image docker build -t sonata-app:latest . # Step 4: Tag image with ECR URL docker tag sonata-app:latest 123456789.dkr.ecr.ap-south-1.amazonaws.com/sonata-app:latest # Step 5: Push image to ECR docker push 123456789.dkr.ecr.ap-south-1.amazonaws.com/sonata-app:latest # Now your image is in ECR! CI/CD — Continuous Integration & Continuous Delivery

Automate everything — from code commit to production deployment, zero manual steps

CI
🔄
Continuous Integration
Every code push → automatically:
1. Pull latest code from Git
2. Compile the project
3. Run all unit tests
4. Run code quality checks
5. Report: ✅ Pass or ❌ Fail

Goal: Catch bugs early, before they reach production!
CD
🚀
Continuous Delivery/Deploy
After CI passes → automatically:
1. Build Docker image
2. Push to ECR
3. Deploy to staging/prod
4. Run smoke tests
5. Notify team (Slack/Email)

Goal: Ship code to production in minutes, not days!
CD*
⚡
Continuous Deployment
No human approval needed!
Every passing commit automatically goes to production.

• Delivery = manual approval before prod
• Deployment = fully automatic to prod

Companies like Netflix deploy 100s of times per day!

GitHub Actions — CI/CD for Your Spring Boot App

Automate build → test → Docker build → push to ECR → deploy on every git push

Where does this file go?

Create file: .github/workflows/deploy.yml in your Spring Boot project root. GitHub automatically detects and runs it on every push!

  .github/workflows/deploy.yml — Full CI/CD Pipeline
# ────────────────────────────────────────────────────────────── # GitHub Actions CI/CD Pipeline for Spring Boot + Docker + AWS ECS # Trigger: Every push to 'main' branch # ────────────────────────────────────────────────────────────── name: Spring Boot CI/CD → AWS ECS on: push: branches: [main] # Runs on every push to main pull_request: branches: [main] # Also runs on PRs (CI only, not deploy) env: # AWS configuration — these come from GitHub Secrets AWS_REGION: ap-south-1 ECR_REGISTRY: 123456789.dkr.ecr.ap-south-1.amazonaws.com ECR_REPOSITORY: sonata-app ECS_SERVICE: sonata-service ECS_CLUSTER: sonata-cluster jobs: # ── JOB 1: CI — Build & Test ────────────────────────────── ci: name: Build & Test runs-on: ubuntu-latest # GitHub gives you a free Ubuntu VM steps: # Step 1: Checkout your code - name: Checkout Code uses: actions/checkout@v4 # Step 2: Set up Java 21 - name: Set up Java 21 uses: actions/setup-java@v4 with: java-version: '21' distribution: 'temurin' # Step 3: Cache Maven dependencies (speeds up builds!) - name: Cache Maven packages uses: actions/cache@v3 with: path: ~/.m2 key: ${{ runner.os }}-maven-${{ hashFiles('**/pom.xml') }} # Step 4: Build and Run Tests - name: Build and Test run: mvn clean verify # Compiles + runs JUnit tests # Step 5: Upload test results as artifact - name: Upload Test Results uses: actions/upload-artifact@v3 if: always() # Upload even if tests fail with: name: test-results path: target/surefire-reports/ # ── JOB 2: CD — Docker Build + ECR Push + ECS Deploy ───── cd: name: Build Docker + Deploy to ECS runs-on: ubuntu-latest needs: ci # Only runs if CI job passes! if: github.ref == 'refs/heads/main' # Only on main branch steps: - name: Checkout Code uses: actions/checkout@v4 - name: Set up Java 21 uses: actions/setup-java@v4 with: java-version: '21' distribution: 'temurin' # Build JAR for Docker image - name: Build JAR run: mvn clean package -DskipTests # Authenticate to AWS using GitHub Secrets - name: Configure AWS Credentials uses: aws-actions/configure-aws-credentials@v4 with: aws-access-key-id: ${{ secrets.AWS_ACCESS_KEY_ID }} aws-secret-access-key: ${{ secrets.AWS_SECRET_ACCESS_KEY }} aws-region: ${{ env.AWS_REGION }} # Login to ECR - name: Login to Amazon ECR id: login-ecr uses: aws-actions/amazon-ecr-login@v2 # Build Docker image and push to ECR - name: Build, Tag & Push Docker Image to ECR id: build-image env: IMAGE_TAG: ${{ github.sha }} # Use git commit SHA as tag run: | # Build Docker image docker build -t $ECR_REGISTRY/$ECR_REPOSITORY:$IMAGE_TAG . docker tag $ECR_REGISTRY/$ECR_REPOSITORY:$IMAGE_TAG $ECR_REGISTRY/$ECR_REPOSITORY:latest # Push to ECR docker push $ECR_REGISTRY/$ECR_REPOSITORY:$IMAGE_TAG docker push $ECR_REGISTRY/$ECR_REPOSITORY:latest # Output image URI for next step echo "image=$ECR_REGISTRY/$ECR_REPOSITORY:$IMAGE_TAG" >> $GITHUB_OUTPUT # Update ECS Service to deploy new container - name: Deploy to Amazon ECS run: | aws ecs update-service \ --cluster $ECS_CLUSTER \ --service $ECS_SERVICE \ --force-new-deployment # ECS pulls new image from ECR and replaces old containers!
Setting GitHub Secrets (Required for AWS)

Go to GitHub repo → Settings → Secrets and Variables → Actions → New Repository Secret
Add: AWS_ACCESS_KEY_ID and AWS_SECRET_ACCESS_KEY (from your AWS IAM user with ECS permissions). Never hardcode credentials in YAML!
