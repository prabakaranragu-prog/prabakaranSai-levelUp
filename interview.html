Java Core — OOP, Collections, Generics

1. Explain the 4 pillars of OOP with Java examples.
Easy Java

1. Encapsulation — Binding data and methods together, hiding internal state via access modifiers.

class BankAccount { private double balance; // hidden public void deposit(double amt) { if(amt > 0) balance += amt; } public double getBalance() { return balance; } }

2. Inheritance — Child class inherits parent class properties. Promotes reuse.

class Animal { void eat() { System.out.println("eating"); } } class Dog extends Animal { void bark() { System.out.println("woof"); } }

3. Polymorphism — Same interface, multiple behaviors. Compile-time (overloading) and Runtime (overriding).

// Runtime polymorphism Animal a = new Dog(); a.eat(); // Dog's eat() called (if overridden)

4. Abstraction — Hiding implementation details, exposing only essential features. Done via abstract classes or interfaces.

interface Payment { void pay(double amount); } class CreditCard implements Payment { public void pay(double amt) { ... } }
Recruiter insight: Always mention the "why" — encapsulation = data security, abstraction = reduce complexity, inheritance = reuse, polymorphism = flexibility.
2. Interface vs Abstract Class — When to use which?
Medium

Abstract Class: Can have state (fields), constructors, concrete methods. Single inheritance only. Use when classes share a common base with shared implementation.

Interface: Multiple inheritance supported. Default/static methods from Java 8. No state (except constants). Use for defining a contract/capability (Serializable, Runnable, Comparable).

// Abstract class — shared state + partial implementation abstract class Vehicle { private String brand; Vehicle(String brand) { this.brand = brand; } abstract void start(); void stop() { System.out.println("stopping"); } // concrete } // Interface — capability contract interface Flyable { void fly(); default void land() { System.out.println("landing"); } } interface Swimmable { void swim(); } class Duck extends Animal implements Flyable, Swimmable { ... }
Rule of thumb: "IS-A" relationship with shared state → abstract class. "CAN-DO" capability → interface. In modern Java (8+), prefer interfaces with default methods.
3. How does HashMap work internally? What happens during hash collision?
Hard Important

Internal structure: Node<K,V>[] array (buckets). Default capacity: 16, load factor: 0.75.

put(key, value) process:

  • Compute hash = key.hashCode() ^ (h >>> 16) (spreading)
  • Find bucket index: hash & (capacity - 1)
  • If bucket empty → insert new Node
  • If collision → traverse linked list, check equals(). If key exists → update value. If not → append.
  • If bucket size > 8 AND table size ≥ 64 → convert to Red-Black Tree (O(log n) lookups)
  • If tree shrinks back to ≤ 6 → converts back to linked list

Resize: When size > capacity × loadFactor → resize to 2× (rehash all entries).

// Java 8 HashMap - simplified int hash = key.hashCode() ^ (key.hashCode() >>> 16); int index = hash & (table.length - 1); // Bucket = linked list until 8 nodes → then Red-Black Tree
Key points to mention: hashCode() + equals() contract, why 0.75 load factor (balances space/time), Java 8 treeification improvement, thread-unsafe (use ConcurrentHashMap for concurrency).
4. HashMap vs LinkedHashMap vs TreeMap vs ConcurrentHashMap
Medium
  • HashMap — No ordering, O(1) avg, null key/values allowed, NOT thread-safe
  • LinkedHashMap — Maintains insertion order (or access order), O(1), null allowed, NOT thread-safe. Good for LRU cache (new LinkedHashMap(cap, 0.75f, true))
  • TreeMap — Sorted by natural order or Comparator, O(log n), NO null keys, NOT thread-safe
  • ConcurrentHashMap — Thread-safe, NO null keys/values. Java 8: uses CAS + synchronized at bucket level (16 default segments in Java 7). High concurrency, good for read-heavy workloads
  • WeakHashMap — Keys held by weak references (GC-able). Good for cache entries
// LRU Cache using LinkedHashMap class LRUCache extends LinkedHashMap<Integer, Integer> { private int capacity; LRUCache(int cap) { super(cap, 0.75f, true); this.capacity = cap; } @Override protected boolean removeEldestEntry(Map.Entry<?,?> e) { return size() > capacity; } }
5. Comparable vs Comparator — Difference and usage
Easy

Comparable — Defines natural ordering inside the class itself. Implements compareTo(T o). One sort sequence.

Comparator — External comparator, separate from class. Multiple sort sequences possible. Functional interface — use lambdas.

// Comparable - natural order (inside class) class Employee implements Comparable<Employee> { String name; int salary; @Override public int compareTo(Employee o) { return this.salary - o.salary; } } Collections.sort(employees); // uses compareTo // Comparator - multiple orderings (external) Comparator<Employee> byName = Comparator.comparing(e -> e.name); Comparator<Employee> bySalary = Comparator.comparingInt(e -> e.salary); Comparator<Employee> complex = Comparator.comparing(Employee::getName) .thenComparingInt(Employee::getSalary) .reversed(); employees.sort(byName); employees.sort(complex);
Prefer Comparator for flexibility. Use Comparator.comparing() — it's null-safe and chaining-friendly.
6. How to create an Immutable class in Java?
Medium

Rules for creating an immutable class:

  • Declare class as final (prevent subclassing)
  • All fields private final
  • No setters
  • Initialize all fields via constructor
  • Deep copy of mutable fields (e.g., collections, dates)
  • Return defensive copies from getters for mutable fields
public final class ImmutableEmployee { private final String name; private final int salary; private final List<String> skills; // mutable field! public ImmutableEmployee(String name, int salary, List<String> skills) { this.name = name; this.salary = salary; this.skills = new ArrayList<>(skills); // defensive copy } public String getName() { return name; } public int getSalary() { return salary; } public List<String> getSkills() { return Collections.unmodifiableList(skills); // defensive copy } }
Java's record keyword (Java 16+) creates immutable classes automatically! Also String, Integer, LocalDate are immutable.
7. Exception Handling — Checked vs Unchecked, Custom Exceptions, Best Practices
Medium

Checked Exceptions — Must be declared or caught. Compiler enforces. Subclass of Exception (not RuntimeException). Example: IOException, SQLException.

Unchecked Exceptions — Subclass of RuntimeException. Not enforced. Example: NullPointerException, IllegalArgumentException.

Error — JVM-level problems (OutOfMemoryError, StackOverflowError). Don't catch these.

// Custom exception hierarchy public class AppException extends RuntimeException { private final String errorCode; public AppException(String errorCode, String message) { super(message); this.errorCode = errorCode; } public AppException(String errorCode, String message, Throwable cause) { super(message, cause); this.errorCode = errorCode; } public String getErrorCode() { return errorCode; } } public class ResourceNotFoundException extends AppException { public ResourceNotFoundException(String id) { super("RESOURCE_NOT_FOUND", "Resource not found: " + id); } } // Best practices try (InputStream in = new FileInputStream("file.txt")) { // try-with-resources // auto-closes } catch (IOException e) { log.error("Failed to read file", e); // log with cause throw new AppException("FILE_READ_FAILED", "Failed to read file", e); // wrap }
Best practices: Prefer unchecked for programming errors. Never catch Exception/Throwable silently. Always log the cause. Use try-with-resources. Don't use exceptions for flow control.
8. SOLID Principles with Java examples
Hard Important
  • S — Single Responsibility: A class should have only one reason to change. Separate UserService from EmailService.
  • O — Open/Closed: Open for extension, closed for modification. Use Strategy pattern, not if-else chains.
  • L — Liskov Substitution: Subclasses must be substitutable for their base classes without altering correctness.
  • I — Interface Segregation: Clients should not be forced to depend on interfaces they don't use. Split fat interfaces.
  • D — Dependency Inversion: High-level modules shouldn't depend on low-level. Both should depend on abstractions (inject interfaces, not concrete classes).

JVM Architecture, GC & Concurrency

9. Explain JVM Memory Model — Heap, Stack, Metaspace
Hard Important
  • Heap — Shared across threads. All objects live here. Divided into: Young Gen (Eden + Survivor S0/S1) and Old Gen (Tenured).
  • Young Gen (Eden) — New objects allocated here. Minor GC (fast) clears it. Survivors promoted to S0/S1.
  • Old Gen — Long-lived objects. Major GC (slow, STW) clears it. Tenuring threshold controls promotion.
  • Metaspace (Java 8+) — Class metadata, method bytecode, static variables. Replaced PermGen. Grows dynamically (native memory). Can cause native OOM.
  • Stack — Per-thread. Method call frames (local variables, operand stack, return address). StackOverflowError = too deep recursion.
  • PC Register — Per-thread. Points to current instruction.
  • Native Method Stack — JNI calls.
// JVM flags for tuning -Xms512m -Xmx2g // initial/max heap -XX:MetaspaceSize=256m // initial metaspace -XX:MaxMetaspaceSize=512m -XX:+UseG1GC // use G1 garbage collector -XX:NewRatio=3 // Old:Young = 3:1
OOM Types: java.lang.OutOfMemoryError: Java heap space → heap full. ...GC overhead limit exceeded → GC running >98% time. ...Metaspace → class loading issue (ClassLoader leak).
10. Garbage Collection — Types of GC algorithms (G1, ZGC, etc.)
Hard
  • Serial GC — Single-threaded. Good for small apps / single CPU.
  • Parallel GC (Throughput) — Multi-threaded. Stop-the-world. Default in Java 8. Good for batch processing.
  • CMS (Concurrent Mark Sweep) — Deprecated Java 9, removed 14. Concurrent collection, short STW pauses.
  • G1 GC (Java 9+ default) — Divides heap into equal-sized regions. Concurrent + incremental. Predictable pause targets. Replaces CMS.
  • ZGC (Java 15+ production) — Sub-millisecond pauses. Scalable to terabytes. Almost fully concurrent. Use -XX:+UseZGC.
  • Shenandoah — RedHat's ultra-low-pause GC. Similar to ZGC.

GC Phases: Mark (find live objects) → Sweep (reclaim dead) → Compact (defragment).

Choose: G1 for most apps. ZGC for latency-sensitive (trading, gaming). Parallel for maximum throughput. Always measure, never guess.
11. volatile vs synchronized vs AtomicInteger — When to use what?
Hard Important

volatile — Guarantees visibility only (changes visible to all threads immediately). Does NOT guarantee atomicity. Use for: simple boolean flags, double-checked locking pattern.

synchronized — Guarantees visibility + atomicity + mutual exclusion. Only one thread at a time. High performance overhead. Use for compound operations.

Atomic classes — Lock-free thread safety using CAS (Compare-And-Swap) CPU instructions. AtomicInteger, AtomicLong, AtomicReference. Much faster than synchronized for single variables.

// volatile - visibility only volatile boolean running = true; // flag pattern void stop() { running = false; } // synchronized - mutual exclusion class Counter { private int count = 0; synchronized void increment() { count++; } // atomic } // AtomicInteger - lock-free, faster AtomicInteger counter = new AtomicInteger(0); counter.incrementAndGet(); // CAS-based counter.compareAndSet(expected, newValue); // ReentrantLock - explicit lock, more control ReentrantLock lock = new ReentrantLock(); lock.lock(); try { /* critical section */ } finally { lock.unlock(); } // ALWAYS unlock in finally!
volatile i++ is NOT atomic (read-modify-write). Use AtomicInteger. Use synchronized/ReentrantLock only when you need compound operations or waiting/notification.

18. @Component vs @Service vs @Repository vs @Controller
Easy Spring

All are stereotype annotations that register beans. They differ in semantic meaning and additional behavior:

  • @Component — Generic stereotype. Use for utility classes, helpers.
  • @Service — Business logic layer. Same as @Component but semantically meaningful. Transactions applied here.
  • @Repository — Data access layer. Additionally: translates persistence exceptions (e.g., SQL exceptions) to Spring's DataAccessException hierarchy.
  • @Controller — Web MVC controller. Processes HTTP requests, returns view names. Used with @RequestMapping.
  • @RestController — @Controller + @ResponseBody. Returns JSON/XML directly. Used for REST APIs.
Key difference: @Repository adds exception translation via PersistenceExceptionTranslationPostProcessor. @RestController vs @Controller: @RestController skips view resolution, writes directly to HTTP response.
19. Spring AOP — Aspect, Advice, Pointcut, JoinPoint, Proxy
Hard Spring
  • Aspect — The module encapsulating cross-cutting concern (logging, security, transactions)
  • Advice — Action to take: @Before, @After, @AfterReturning, @AfterThrowing, @Around
  • Pointcut — Expression selecting which JoinPoints to intercept. e.g., execution(* com.example.service.*.*(..))
  • JoinPoint — Point in execution (method call, field access). Spring AOP only supports method execution.
  • Proxy — Spring creates JDK dynamic proxy (for interface) or CGLIB proxy (for class). Wraps the bean.
@Aspect @Component public class LoggingAspect { @Pointcut("execution(* com.example.service..*(..))") private void serviceLayer() {} @Before("serviceLayer()") public void logBefore(JoinPoint jp) { log.info("Calling: {}", jp.getSignature().getName()); } @Around("@annotation(Auditable)") // annotation-based pointcut public Object audit(ProceedingJoinPoint pjp) throws Throwable { long start = System.currentTimeMillis(); try { Object result = pjp.proceed(); // call actual method log.info("Completed in {}ms", System.currentTimeMillis() - start); return result; } catch (Exception e) { log.error("Failed: {}", pjp.getSignature(), e); throw e; } } }
AOP Limitations: Only works on Spring-managed beans. Self-invocation bypasses proxy (use AopContext.currentProxy()). Only method-level joinpoints. CGLIB proxies require no-arg constructor.
20. @Transactional — Propagation, Isolation, Rollback, Pitfalls
Hard Important

Propagation types:

  • REQUIRED (default) — Join existing tx or create new
  • REQUIRES_NEW — Always create new tx. Suspend existing.
  • SUPPORTS — Use existing if available, else non-transactional
  • MANDATORY — Must exist, else throw exception
  • NEVER — Must NOT be in transaction
  • NESTED — Nested tx using savepoints

Isolation levels: READ_UNCOMMITTED, READ_COMMITTED, REPEATABLE_READ, SERIALIZABLE

@Transactional( propagation = Propagation.REQUIRED, isolation = Isolation.READ_COMMITTED, rollbackFor = {Exception.class}, // rollback on checked too noRollbackFor = {InventoryException.class}, readOnly = true, // optimization for SELECT timeout = 30 ) public Order createOrder(OrderRequest req) { ... } // PITFALL 1: self-invocation bypasses proxy @Service class OrderService { void outer() { inner(); } // WRONG - no transaction! @Transactional void inner() { ... } // not invoked through proxy } // PITFALL 2: @Transactional on private methods - doesn't work! // PITFALL 3: RuntimeException rolls back, checked Exception does NOT by default // Use rollbackFor = Exception.class for checked exceptions

31. Stream API — All Intermediate & Terminal Operations with examples
Hard Important

Intermediate Operations — Lazy; return a Stream. Chain as many as needed.

  • filter(Predicate) — Keep elements matching condition
  • map(Function) — Transform each element (1-to-1)
  • flatMap(Function) — Transform and flatten (1-to-many)
  • distinct() — Remove duplicates (uses equals/hashCode)
  • sorted() / sorted(Comparator) — Natural or custom order
  • peek(Consumer) — Debug/log without modifying (intermediate)
  • limit(n) — Take first n elements
  • skip(n) — Skip first n elements
  • mapToInt / mapToLong / mapToDouble — Specialized primitive streams

Terminal Operations — Trigger the pipeline; return a result.

  • collect(Collector) — Gather into collection/map/string
  • forEach(Consumer) — Iterate (no result)
  • reduce(identity, BinaryOperator) — Fold to single value
  • count() — Count elements (long)
  • findFirst() / findAny() — Return Optional (first or any)
  • anyMatch / allMatch / noneMatch(Predicate) — Boolean checks
  • min / max(Comparator) — Return Optional min/max
  • toArray() — Convert to array
List<Employee> employees = List.of( new Employee("Alice", "IT", 85000), new Employee("Bob", "HR", 55000), new Employee("Carol", "IT", 92000), new Employee("Dave", "HR", 61000), new Employee("Eve", "IT", 78000) ); // filter + map + collect List<String> itNames = employees.stream() .filter(e -> e.getDept().equals("IT")) .map(Employee::getName) .sorted() .collect(Collectors.toList()); // [Alice, Carol, Eve] // reduce - sum salaries double totalSalary = employees.stream() .mapToDouble(Employee::getSalary) .sum(); // or .reduce(0, Double::sum) // peek - debug List<String> result = employees.stream() .filter(e -> e.getSalary() > 60000) .peek(e -> System.out.println("After filter: " + e.getName())) .map(Employee::getName) .peek(n -> System.out.println("After map: " + n)) .collect(Collectors.toList()); // findFirst with Optional Optional<Employee> topEarner = employees.stream() .max(Comparator.comparingDouble(Employee::getSalary)); topEarner.ifPresent(e -> System.out.println("Top: " + e.getName())); // Carol // count + anyMatch long itCount = employees.stream().filter(e -> "IT".equals(e.getDept())).count(); // 3 boolean anyHighEarner = employees.stream().anyMatch(e -> e.getSalary() > 90000); // true
Key Rule: Streams are lazy — intermediate ops only run when terminal op is called. A stream can only be used ONCE. Reuse by calling .stream() again on the source collection.

33. flatMap vs map — Difference with real examples (String split, nested lists)
Medium Important

map() — 1-to-1 transform. Each element maps to exactly ONE output. Returns Stream<Stream<T>> if function returns a Stream.

flatMap() — 1-to-many transform. Each element maps to a STREAM of outputs, then all streams are FLATTENED into one. Returns Stream<T>.

// PROBLEM: Split sentences into individual words List<String> sentences = List.of("Hello World", "Java Stream API", "FlatMap is powerful"); // map() - gives Stream<String[]> - WRONG for most use cases Stream<String[]> wrong = sentences.stream().map(s -> s.split(" ")); // result: [["Hello","World"], ["Java","Stream","API"], ...] ← nested! // flatMap() - gives Stream<String> - CORRECT, flattened List<String> words = sentences.stream() .flatMap(s -> Arrays.stream(s.split(" "))) .distinct() .sorted() .collect(Collectors.toList()); // [API, FlatMap, Hello, Java, Stream, World, is, powerful] // ==================================================== // NESTED LISTS example List<List<Integer>> nested = List.of( List.of(1, 2, 3), List.of(4, 5), List.of(6, 7, 8, 9) ); // map - gives Stream<List<Integer>> List<List<Integer>> mapResult = nested.stream() .map(list -> list) .collect(Collectors.toList()); // [[1,2,3],[4,5],[6,7,8,9]] // flatMap - flattens to Stream<Integer> List<Integer> flatResult = nested.stream() .flatMap(Collection::stream) .collect(Collectors.toList()); // [1,2,3,4,5,6,7,8,9] // ==================================================== // Optional flatMap - avoid nested Optional<Optional<T>> Optional<String> opt = Optional.of(" hello "); Optional<String> trimmed = opt.map(String::trim); // Optional<String> Optional<Optional<String>> bad = opt.map(s -> Optional.of(s.trim())); // nested! Optional<String> good = opt.flatMap(s -> Optional.of(s.trim())); // flat Optional
Rule of thumb: If your map() lambda returns a Stream or Optional, use flatMap() instead to avoid nesting. FlatMap = map + flatten in one operation.

35. parallelStream() — When to use, pitfalls, and thread safety
Medium

parallelStream() uses ForkJoinPool (common pool, default = CPU cores - 1) to process elements in parallel.

Use when: Large data set (>10k elements), CPU-intensive operations (no I/O), order doesn't matter, elements are independent.

Avoid when: Small collections (overhead > benefit), I/O operations, shared mutable state, ordered output required.

// ✅ Good use case: CPU-intensive on large collection List<Double> result = largeList.parallelStream() .map(n -> expensiveCalculation(n)) // CPU-bound .filter(n -> n > 0) .collect(Collectors.toList()); // ❌ WRONG: Shared mutable state - race condition! List<Integer> unsafeList = new ArrayList<>(); numbers.parallelStream() .forEach(n -> unsafeList.add(n)); // ArrayList NOT thread-safe! Data loss! // ✅ CORRECT: Use thread-safe collection or collect() List<Integer> safeResult = numbers.parallelStream() .filter(n -> n > 0) .collect(Collectors.toList()); // collect() is thread-safe // ❌ WRONG: parallelStream with I/O (blocks ForkJoinPool threads) employees.parallelStream() .forEach(e -> emailService.sendEmail(e)); // Bad! I/O in ForkJoin // ✅ CORRECT: Use CompletableFuture for I/O tasks List<CompletableFuture<Void>> futures = employees.stream() .map(e -> CompletableFuture.runAsync(() -> emailService.sendEmail(e), customExecutor)) .collect(Collectors.toList()); CompletableFuture.allOf(futures.toArray(new CompletableFuture[0])).join(); // Custom thread pool for parallelStream (avoid using common pool) ForkJoinPool customPool = new ForkJoinPool(4); List<Integer> customResult = customPool.submit(() -> numbers.parallelStream() .filter(n -> n % 2 == 0) .collect(Collectors.toList()) ).get();
Interview answer: "parallelStream() is not always faster. For small collections, sequential stream wins. I benchmark with JMH before switching. I use it only for CPU-bound, stateless, order-independent operations on large datasets."
36. Optional — Complete Guide: orElse vs orElseGet, map, flatMap, filter, ifPresent
Medium Important

Optional is a container that may or may not contain a value. Eliminates null checks and NullPointerException.

// Creating Optionals Optional<String> empty = Optional.empty(); Optional<String> present = Optional.of("hello"); // throws NPE if null Optional<String> nullable = Optional.ofNullable(null); // safe with null // ---- orElse vs orElseGet ---- // orElse - ALWAYS evaluates the argument (even if value present!) String val1 = present.orElse(expensiveDefaultMethod()); // method ALWAYS called // orElseGet - LAZY, only evaluated if empty String val2 = present.orElseGet(() -> expensiveDefaultMethod()); // method NOT called if present // ✅ PREFER orElseGet for expensive defaults! // orElseThrow String val3 = present.orElseThrow(() -> new ResourceNotFoundException("not found")); // ---- map vs flatMap ---- Optional<String> name = Optional.of(" Prabakaran "); Optional<String> upper = name.map(String::trim).map(String::toUpperCase); // "PRABAKARAN" // flatMap - when function returns Optional (avoid Optional<Optional<>>) Optional<User> user = findUser(1L); Optional<Address> address = user.flatMap(User::getAddress); // not user.map(u -> u.getAddress()) // ---- filter ---- Optional<Integer> evenNumber = Optional.of(42) .filter(n -> n % 2 == 0); // Optional[42] Optional<Integer> filteredOut = Optional.of(41) .filter(n -> n % 2 == 0); // Optional.empty // ---- ifPresent vs ifPresentOrElse (Java 9) ---- user.ifPresent(u -> System.out.println("Found: " + u.getName())); user.ifPresentOrElse( u -> System.out.println("Found: " + u.getName()), () -> System.out.println("User not found") ); // ---- Chaining multiple nullable calls (avoids NPE chain) ---- // Old way (NPE-prone) // String city = user.getAddress().getCity().getName(); // 3 potential NPEs // New way with Optional String city = Optional.ofNullable(user) .map(User::getAddress) .map(Address::getCity) .map(City::getName) .orElse("Unknown");
Don't: Never use Optional for method parameters or fields — it's meant for return types only. Don't call .get() without isPresent() check. Prefer orElseGet() over orElse() for costly defaults.

41. Singleton Pattern — All thread-safe implementations explained
Hard Most Asked

Singleton ensures only ONE instance exists in the entire application. Spring beans are Singleton by default!

// ❌ NOT thread-safe (classic) public class Singleton { private static Singleton instance; private Singleton() {} public static Singleton getInstance() { if (instance == null) { // Two threads can pass this check simultaneously! instance = new Singleton(); } return instance; } } // ✅ Thread-safe: Synchronized (slow - lock on every call) public class SynchronizedSingleton { private static SynchronizedSingleton instance; private SynchronizedSingleton() {} public static synchronized SynchronizedSingleton getInstance() { if (instance == null) instance = new SynchronizedSingleton(); return instance; } } // ✅ Thread-safe: Double-Checked Locking (recommended for lazy init) public class DCLSingleton { private static volatile DCLSingleton instance; // volatile is CRITICAL! private DCLSingleton() {} public static DCLSingleton getInstance() { if (instance == null) { // 1st check (no lock) - performance synchronized (DCLSingleton.class) { if (instance == null) { // 2nd check (with lock) - safety instance = new DCLSingleton(); } } } return instance; } } // ✅ Thread-safe: Bill Pugh / Inner static class (BEST - lazy, no sync overhead!) public class BillPughSingleton { private BillPughSingleton() {} // Inner class loaded only when getInstance() is called private static class SingletonHolder { private static final BillPughSingleton INSTANCE = new BillPughSingleton(); } public static BillPughSingleton getInstance() { return SingletonHolder.INSTANCE; } } // ✅ Thread-safe: Enum (BEST for preventing reflection/serialization attacks!) public enum EnumSingleton { INSTANCE; public void doSomething() { System.out.println("Enum singleton!"); } } // Usage: EnumSingleton.INSTANCE.doSomething(); // ✅ Spring way (managed by IoC container) @Service // or @Component — Spring handles singleton lifecycle public class SpringSingleton { // Spring creates ONE instance per ApplicationContext }
Interview ranking: Enum > Bill Pugh > DCL > Synchronized. Enum prevents reflection attacks and serialization issues. Spring beans ARE singletons managed by Spring — mention this! volatile in DCL prevents instruction reordering/partial construction.

43. Strategy Pattern — Eliminate if-else with Spring DI injection

Answer this in layers (impress the interviewer!):

46. Production app shows OutOfMemoryError — how do you debug and fix it?
Hard Real-World

Types of OOM Errors:

  • Java heap space — Too many objects in heap. Memory leak or heap too small.
  • GC overhead limit exceeded — GC running >98% time but reclaiming <2% memory.
  • Metaspace — Too many class definitions loaded. ClassLoader leak.
  • Unable to create native thread — Too many threads. Thread leak.

Debugging Steps:

  1. Enable heap dump on OOM: Add -XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/tmp/heapdump.hprof
  2. Analyze heap dump: Eclipse MAT (Memory Analyzer Tool) or VisualVM → find largest objects
  3. Check GC logs: -verbose:gc -Xlog:gc*:gc.log
  4. Monitor with Actuator: /actuator/metrics/jvm.memory.used
  5. Use APM: New Relic, Dynatrace, or Prometheus + Grafana for trends

Common Root Causes:

  • Static collections growing unbounded (cache without eviction)
  • Not closing resources (InputStream, Connection not in try-with-resources)
  • Event listeners not unregistered (memory retained)
  • Incorrect JPA fetch (loading entire table into memory)
  • String concatenation in loops (use StringBuilder)
// ❌ Memory leak - static Map growing unbounded public class CacheService { private static final Map<String, Object> cache = new HashMap<>(); // NEVER evicts! public void put(String key, Object val) { cache.put(key, val); } } // ✅ Fix - Use Caffeine cache with size limit and TTL @Bean CacheManager cacheManager() { return new CaffeineCacheManager() {{ setCaffeine(Caffeine.newBuilder() .maximumSize(1000) .expireAfterWrite(30, TimeUnit.MINUTES) .recordStats()); }}; } // ❌ Memory leak - not closing resources InputStream is = new FileInputStream("file.txt"); // reading... but never closed! Kept in memory // ✅ Fix - try-with-resources try (InputStream is = new FileInputStream("file.txt"); BufferedReader reader = new BufferedReader(new InputStreamReader(is))) { return reader.lines().collect(Collectors.joining("\n")); } // auto-closed! // ❌ Loading entire table into memory List<Order> allOrders = orderRepo.findAll(); // MILLIONS of records → OOM! // ✅ Fix - Use pagination or streaming // Option 1: Paginated Page<Order> page = orderRepo.findAll(PageRequest.of(0, 100)); // Option 2: Stream with @Transactional @Transactional(readOnly = true) public void processAllOrders() { orderRepo.streamAll().forEach(order -> process(order)); // JPA stream } // JVM flags for production // -Xms1g -Xmx4g heap // -XX:+HeapDumpOnOutOfMemoryError auto heap dump // -XX:HeapDumpPath=/tmp/heapdump.hprof // -XX:+UseG1GC G1 GC // -XX:MaxGCPauseMillis=200 target pause
Interview answer structure: "First I'd check the OOM type from the error message, enable heap dump on OOM, analyze with Eclipse MAT to find the dominator tree, then trace back to the root cause. Common fixes: add eviction to caches, use try-with-resources, paginate large queries."

Q2. What is the difference between a Docker Image and a Container?
Easy
  • Image = read-only blueprint/template built from Dockerfile. Stored in registry. Like a Java Class definition.
  • Container = running instance of an image. Created by docker run. Like a Java Object (new MyApp()). You can run 10 containers from 1 image independently.
Java Analogy: Image = Class. Container = Object. One image, many running containers.

  • IAM -- Access management. ECS Task Role allows container to access S3/RDS/Secrets -- no hardcoded keys!
  • Secrets Manager -- Store DB passwords. Spring Boot fetches at runtime via AWS SDK.
  • CloudWatch -- Logs, metrics, alarms. View Spring Boot logs, set CPU/error alerts.
  • SQS -- Message queue. Decouple microservices asynchronously.
Q7. EC2 vs ECS vs Lambda -- when to use each?
Medium
  • EC2 -- Full virtual server. You manage OS, patches, Java. Use when you need full control or custom AMI. Pay per hour (even when idle).
  • ECS Fargate -- Serverless containers. Provide Docker image, AWS handles the server underneath. Best for Spring Boot APIs -- auto-scales, no server management. Pay per CPU/RAM used.
  • Lambda -- Serverless functions. Event-driven, short-lived. Spring Boot works but cold start is an issue. Best for: S3 events, SQS consumers, scheduled tasks. Pay per invocation.
For Spring Boot REST APIs: Always recommend ECS Fargate -- containerized, portable, auto-scaling, no server management overhead.
Q8. How do you pass DB credentials to Spring Boot running in Docker/ECS securely?
Hard

NEVER hardcode in application.properties or Dockerfile!

Approach 1 -- Environment Variables:

# application.properties spring.datasource.url=${DB_URL} spring.datasource.password=${DB_PASSWORD} # Pass via docker-compose or ECS Task Definition environment section

Approach 2 -- AWS Secrets Manager (Best for prod):

# Spring Cloud AWS reads secrets automatically spring.config.import=aws-secretsmanager:/myapp/prod/db # AWS stores: {"DB_PASSWORD":"secret123","DB_USER":"admin"} # Spring Boot injects into properties at startup!
In ECS: Add secrets to Task Definition. ECS injects as env vars at container start. Combined with IAM Task Role -- zero credentials in code!
Q9. What is a Docker Volume? Why is it needed?
Easy

Containers are stateless -- all data lost when container stops. Volume = persistent storage outside the container lifecycle.

  • MySQL container -- Mount volume at /var/lib/mysql. Data persists after restart!
  • Bind Mount -- Map host directory to container. e.g., -v /host/logs:/app/logs
  • Named Volume -- Docker manages it. Best for production databases.
docker run -v mysql-data:/var/lib/mysql mysql:8.0 # mysql-data is a named volume -- data survives container removal!
Q10. How do containers communicate with each other? What is Docker Network?
Medium

Docker Network is a virtual network allowing containers to talk using service names as hostnames.

In docker-compose, all services join same default network. Spring Boot reaches MySQL at mysql-db:3306 -- no IP needed!

# Without Docker network -- fragile, IP changes on restart! spring.datasource.url=jdbc:mysql://172.17.0.2:3306/db # With Docker network -- stable, uses service name spring.datasource.url=jdbc:mysql://mysql-db:3306/db
Q11. What is IAM in AWS? Explain Users vs Roles vs Policies
Medium
  • IAM User -- Represents a person. Has long-term Access Keys. Used for AWS Console/CLI.
  • IAM Role -- Represents an AWS service. No long-term keys -- temporary tokens. ECS Task Role lets your container access S3/Secrets Manager without any keys in code!
  • IAM Policy -- JSON permission document. Attached to User or Role. e.g., AmazonS3ReadOnlyAccess.
Principle of Least Privilege: ECS container Role should only have exactly what it needs -- S3:GetObject, SecretsManager:GetSecretValue. Not admin!
Q12. What is a VPC? Explain Public vs Private subnets
Medium

VPC = your own isolated private network in AWS.

  • Public Subnet -- Has route to Internet Gateway. Resources get public IPs. Put: ALB, NAT Gateway here.
  • Private Subnet -- No direct internet. Resources go out via NAT but cannot be accessed from internet. Put: ECS containers, RDS here!

Best practice architecture: ALB (public) receives internet traffic, forwards to ECS (private), ECS connects to RDS (private). DB is never exposed to internet!

Q13. How do you achieve zero-downtime deployment with Docker + ECS?
Hard

ECS Rolling Update:

  1. New task definition created with new Docker image
  2. ECS starts new container
  3. ALB health check: GET /actuator/health returns 200
  4. ALB routes traffic to new container
  5. Old container gracefully drained (in-flight requests complete)
  6. Old container stopped -- zero downtime!
# application.properties -- enable graceful shutdown server.shutdown=graceful spring.lifecycle.timeout-per-shutdown-phase=30s # Spring Boot finishes current requests before stopping!
Run 2+ tasks always! Desired count=2, multi-AZ. If 1 crashes, second serves traffic. No single point of failure.
Q14. How do you monitor a Spring Boot app deployed on AWS?
Medium
  • Logs -- Spring Boot logs sent to AWS CloudWatch Logs via ECS awslogs driver. Query with CloudWatch Insights.
  • Health -- ALB pings /actuator/health every 30s. Unhealthy = container replaced automatically.
  • Metrics -- Actuator + Micrometer exports JVM memory, GC, HTTP latency, thread count to CloudWatch.
  • Alarms -- CPU over 80%, error rate over 5% triggers SNS notification to Slack/Email.
  • Tracing -- AWS X-Ray or Zipkin for distributed tracing across microservices.
# application.properties management.endpoints.web.exposure.include=health,metrics,info management.endpoint.health.show-details=always
Q15. How do you handle dev/staging/prod environments in Docker + Spring Boot?
Hard

Same Docker image for all environments -- inject config via env vars. Never build separate images per env!

  • application-dev.properties -- local DB, DEBUG logs
  • application-staging.properties -- staging DB, INFO logs
  • application-prod.properties -- RDS, WARN logs, tuning
# Activate via env var in ECS Task Definition SPRING_PROFILES_ACTIVE=prod # docker-compose.yml for local dev environment: - SPRING_PROFILES_ACTIVE=dev - DB_URL=jdbc:mysql://mysql:3306/devdb
One image, different configs. In prod: configs from Secrets Manager + env vars. In dev: docker-compose with .env file. Never rebuild image for config changes!
